## 1. 调研概览

本次调研共覆盖 **15 款** 蜜罐产品，涵盖三个类别：

| 类别 | 产品 | 数量 |
|------|------|------|
| 国际开源蜜罐 | T-Pot, Cowrie, Dionaea, Honeyd, Conpot, Glutton, OpenCanary, Beelzebub, Galah | 9 |
| 国产开源蜜罐 | HFish, DecoyMini | 2 |
| 商业蜜罐方案 | Thinkst Canary, FortiDeceptor, 创宇蜜罐, 长亭谛听(D-Sensor), 默安幻阵, SecurityHive | 6 |

> 注：T-Pot 作为集成平台同时包含了 Cowrie、Dionaea、Conpot、Galah、Beelzebub 等子蜜罐，统计时有重复计入。

## 2. 产品详细分析

### 2.1 Cowrie

| 维度 | 详情 |
|------|------|
| **项目地址** | https://github.com/cowrie/cowrie |
| **类型** | 中高交互蜜罐 |
| **开发语言** | Python (Twisted) |
| **许可证** | BSD-3-Clause |
| **Stars** | ~6,400+ |
| **最新活跃** | 2026年5月（持续活跃） |
| **协议模拟** | SSH, Telnet |
| **核心能力** | 模拟完整 UNIX 系统环境（文件系统、命令执行），支持交互式 shell 会话记录、暴力破解日志、恶意文件下载捕获。v3.x 新增 LLM 模式，使用大语言模型动态生成攻击者命令响应，实现"伪高交互"体验 |
| **欺骗能力** | 极高——可配置虚假文件系统（/etc/passwd, /etc/shadow 等），模拟 Debian 12 操作系统指纹，支持自定义 honeyfs |
| **告警集成** | JSON 日志输出、Elasticsearch、hpfeeds、Cuckoo Sandbox、VirusTotal、Malshare 联动 |
| **部署架构** | 单节点，支持 Docker 部署。通过 proxy 模式可代理到真实系统实现高交互 |
| **资源占用** | 轻量级，CPU <5%（空闲状态），内存约 100-200MB |
| **扩展性** | 插件式输出模块（Output Plugins），支持自定义命令模拟 |
| **独特优势** | ① LLM 驱动的动态命令响应（Cowrie v3.x 引入）② 业界最成熟的 SSH 蜜罐，社区最大 ③ 支持 Telnet 协议 |
| **不足** | 协议覆盖少（仅 SSH/Telnet）；无原生 Web 管理界面；不擅长非 SSH 场景的恶意软件捕获 |

### 2.2 Dionaea

| 维度 | 详情 |
|------|------|
| **项目地址** | https://github.com/DinoTools/dionaea |
| **类型** | 低交互蜜罐 |
| **开发语言** | C/Python（Python 3 版本） |
| **许可证** | GPL-2.0 |
| **Stars** | ~800+ |
| **最新活跃** | 中等活跃度 |
| **协议模拟** | FTP, HTTP, Memcache, Mirror, MSSQL, MySQL, PPTP, SIP, SMB, TFTP, UPnP, epmap |
| **核心能力** | 专注于**恶意软件捕获**。通过 libemu 检测 Shellcode 并提取攻击 payload，自动保存二进制样本。不仅是简单的协议模拟，而是构造漏洞响应来诱骗攻击者投递恶意载荷 |
| **欺骗能力** | 中等——协议仿真深度有限（低交互），但覆盖广度极佳（12+ 协议）。在恶意软件诱捕方面表现突出 |
| **告警集成** | hpfeeds 日志推送、JSON 输出、支持 DionaeaFR 前端可视化 |
| **部署架构** | 单节点，Docker 支持良好 |
| **资源占用** | CPU 15-30%，内存 200-500MB（协议较多时） |
| **扩展性** | 模块化设计（C 模块 + Python 脚本），新增协议需 C 编码，门槛较高 |
| **独特优势** | ① 最强的恶意软件自动捕获能力 ② 覆盖 12+ 种协议 ③ 可提取 Shellcode 并联动沙箱分析 |
| **不足** | 无 Web 管理后台；协议仿真深度不够容易指纹识别；Python 3 版本维护不如 v2（C 版本）稳定 |

| **不足** | 资源需求极高（16GB RAM），不适合轻量级部署；组件众多，调试复杂；内核依赖 Linux |

| 维度 | 详情 |
|------|------|

| **独特优势** | ① 协议无关架构（不需要预先绑定端口）② 极低资源占用 ③ 全端口自动嗅探 |
| **不足** | 协议仿真深度不够；依赖 iptables TPROXY（仅 Linux）；缺乏可视化界面 |

### 3.2 Laji-HoneyPot 定位分析

基于对 Laji-HoneyPot 现有代码的分析，其当前优势与不足如下：

**优势：**

- 协议覆盖少于 T-Pot（20+）、HFish（40+）、Dionaea（12+）

- 缺少蜜饵投放机制（HFish、创宇蜜罐、默安幻阵已有）
- 社区规模和知名度较小
- 无内置威胁情报集成

## 7. 总结与建议

### 7.1 Laji-HoneyPot 的核心定位建议

基于竞品分析，Laji-HoneyPot 的独特竞争优势应聚焦于：

> **"溯源反制型蜜罐"** —— 在保留通用蜜罐能力的基础上，以**攻击者溯源 + 主动反制**为核心差异化特色。

当前市场上：
- **T-Pot** 定位"最全面的蜜罐平台"（广度优先）
- **Cowrie** 定位"最深入的 SSH 蜜罐"（深度优先）
- **HFish** 定位"国产最易用的蜜罐"（易用性优先）
- **长亭谛听/创宇蜜罐/默安幻阵** 定位"企业级商业蜜罐"（商业服务优先）

**Laji-HoneyPot 空白机会**：业内尚无一款**以溯源反制为核心特色**的开源蜜罐产品，Laji-HoneyPot 已在此领域建立了先发优势（截屏、指纹采集、Payload 投递、植入物管理等）。

### 7.2 短期行动建议（30 天内）

1. **LLM 集成**（对 SSH/HTTP 协议引入 LLM 动态响应）——这是 2025-2026 年蜜罐领域最显著的趋势（Cowrie、Beelzebub、Galah、T-Pot 均已跟进）
2. **蜜饵投放系统 MVP**——生成虚假凭证文件并投放至蜜罐文件系统
3. **扩展 3-5 种高频协议**（Telnet、VNC、MongoDB 优先）

### 7.3 中期行动建议（60-90 天）

1. **恶意软件自动捕获**——补齐 Laji-HoneyPot 最大的功能缺口
2. **告警渠道扩展**——钉钉/飞书/企业微信，贴近国内用户习惯
3. **攻击者画像 MVP**——基于已有溯源数据构建攻击者档案

### 7.4 长期愿景

成为 **"最懂攻击者的开源蜜罐"**，不仅捕获攻击行为，更精准识别攻击者身份，实现从"被动诱捕"到"主动溯源"的范式跃迁。
